## **Рішення Команди Makaron4ik23 (Catfish)**

### **1. Склад команди**
* **Команда**: Makaron4ik23 / Catfish
* **Склад**: Автономний розробник Antigravity + команда розробників хакатону.

### **2. Короткий опис підходу**
Наше рішення реалізує повністю децентралізоване, автономне відстеження дрона-лідера за допомогою комп'ютерного зору (CV) та оптичного зв'язку, повністю виключаючи обмін GPS-даними або telemetry-координатами між дронами.
* **Комп'ютерний зір (CV)**: Використовує адаптивні HSV-фільтри для локалізації світлодіодного маяка на хвості дрона-лідера. Додатково застосовує фільтр solidity (заповненості контуру) та обмеження за геометричними розмірами для надійної фільтрації сонячних відблисків та дерев.
* **Слідування та керування**: Реалізовано PD-регулятори для лінійної швидкості (pitch), кутової швидкості (yaw) та вертикальної швидкості (climb rate). Похідна помилки згладжується експоненціальним фільтром низьких частот для усунення високочастотного піксельного шуму.
* **Зміщення (Stagger)**: Застосовується візуальний горизонтальний зсув камери залежно від ID дрона (зигзаг), що забезпечує чисту лінію видимості (line-of-sight) в ланцюжку. Оцінка дистанції при stagger виконується на основі площі світлодіода.
* **Антиколізійна безпека**: Stateful-state machine на базі гістерезису. При критичному зближенні відбувається миттєвий обхід згладжування та екстрене гальмування/рух назад на `-1.5` м/с.
* **Обхід перешкод**: Аналіз щільності ребер Canny в центральній зоні камери (із маскуванням LED-цілі). При наближенні до перешкод дрон знижує швидкість та здійснює бічне зміщення в сторону найменшої щільності перешкоди.
* **Компенсація нахилу камери**: Динамічне коригування вертикального положення цілі на основі поточного кута тангажу (pitch) з MAVSDK-телеметрії.

### **3. Команди запуску**
1. **Запуск симуляції**:
   ```sh
   source /opt/ros/humble/setup.bash
   cd ~/PX4-Autopilot/
   source ~/falcon_gaze/resources/scripts/px4_gz_setup.sh
   ros2 launch ~/falcon_gaze/resources/scripts/swarn_launch.py
   ```
2. **Запуск місії ведених дронів**:
   ```sh
   source /opt/ros/humble/setup.bash
   python3 examples/follower_mission.py
   ```
3. **Запуск тестового польоту лідера**:
   ```sh
   source /opt/ros/humble/setup.bash
   python3 resources/scripts/mission_launch.py
   ```
4. **Запуск автономних юніт-тестів**:
   ```sh
   python3 tests/test_follower_logic.py
   ```
5. **Запуск dev-скрипта калібрування дистанції**:
   ```sh
   source /opt/ros/humble/setup.bash
   python3 scratch/calibrate_stagger.py
   ```

### **4. Використані ROS 2 топіки та сервіси**
* **Camera Subscriber**: `/world/{WORLD_NAME}/model/x500_mono_cam_{id}/link/mono_cam/base_link/sensor/camera_sensor/image` — отримання відеопотоку.
* **LED Publisher**: `/model/x500_mono_cam_{id}/led_cmd` — керування світлодіодом.

### **5. Опис LED-протоколу**
* **FOLLOW (Слідування)**: Постійний зелений LED (duty cycle > 80%).
* **HOLD (Утримання позиції)**: Блимання на частоті 1 Гц (період 1.0с: 0.5с ON, 0.5с OFF). Дроном детектується через 40-кадрове вікно як:
  - `0.35 <= duty_cycle <= 0.65`
  - Кількість переходів станів (edges) у межах від 2 до 5.
* **LAND / FINISH (Посадка)**: Світлодіод вимкнено (duty cycle <= 20%). Дрон-послідовник переходить у стан посадки, завершуючи роботу свого LED, передаючи сигнал ланцюжком далі.

### **6. Що працює та відомі обмеження**
* **Працює**:
  - Точне ланцюжкове слідування один за одним («гуськом») із компенсацією перекриття (зигзаг).
  - Миттєвий аварійний backup без затримок.
  - Компенсація нахилів камери під час прискорень.
  - Автоматичне коригування стартових висот відповідно до рельєфу (TARGET_ABS_ALT = 42.0m).
  - Обхід перешкод за допомогою аналізу ребер.
* **Відомі обмеження та невирішені проблеми**:
  1. **Накопичення затримок у довгих ланцюжках (ефект "рогатки")**: Через затримку обробки кадрів та реакцію PD-контролера, коливання швидкості першого дрона підсилюються для наступних дронів (особливо для 3-го послідовника), що може створювати перевантаження регулятора.
  2. **Втрата LED при різкому розвороті на 180°**: За швидкого повороту лідера LED-маяк виходить із зони видимості камери послідовника, запускаючи yaw-пошук (`YAW_SEARCH`). Це безпечно, але тимчасово перериває пряме слідування.
  3. **Спрощений обхід широких перешкод (Canny edges)**: Алгоритм аналізує щільність ребер у поточному кадрі камери для слайд-ухилення. Якщо перешкода занадто широка або оточує дрон з обох сторін, дрон може не знайти безпечного коридору без повноцінної карти глибини (Depth Map) або сенсорів LiDAR.
  4. **Чутливість HSV до змінних умов освітлення**: Тіні або яскраві зелені текстури в симуляції Gazebo за певних кутів сонця можуть викликати короткочасні шуми детекції, хоча геометрична фільтрація (solidity, розміри) успішно відсікає більшість із них.
